// PART 2 — Safe procedure for re-processing the 847 failed jobs
// ---------------------------------------------------------------

// STEP A — Fix the code FIRST, deploy, restart workers
//   Do NOT retry jobs before the fix is deployed.
//   Retrying 847 broken jobs will simply produce 847 more failures.
//   Sequence:
//     1. Merge and deploy the fixed job class.
//     2. Run: php artisan queue:restart
//        This signals all running workers to finish their current job
//        and exit cleanly. Supervisor or your process manager will
//        relaunch them with the new code.
//     3. Verify one job manually: php artisan tinker
//        Then: ProcessBookingConfirmation::dispatch(Booking::find(1))->onConnection('sync')
//        Confirm it succeeds before touching the 847 failed jobs.

// STEP B — Inspect failed jobs before retrying
//   Run: php artisan queue:failed
//   This lists all failed jobs with their ID, job name, queue, and failure time.
//   Check the "payload" column in the failed_jobs table to see if all 847
//   failures are ProcessBookingConfirmation or if other job types are mixed in.
//   Only retry jobs you understand.

// STEP C — Retry selectively by job class (safest approach)
//   There is no built-in --filter flag on queue:retry, so use a workaround:
//     Query failed_jobs directly in tinker or a custom Artisan command:
//       DB::table('failed_jobs')
//         ->where('payload', 'like', '%ProcessBookingConfirmation%')
//         ->pluck('uuid')
//     Then retry each UUID:
//       php artisan queue:retry {uuid1} {uuid2} ...
//   Or write a small Artisan command that reads the UUIDs and calls
//   Artisan::call('queue:retry', ['id' => $uuid]) in a loop.

// STEP D — Retry all (only when all failed jobs are the same class)
//   If you have confirmed that ALL 847 failures are ProcessBookingConfirmation
//   and you are confident the fix is deployed:
//     php artisan queue:retry all
//   This re-queues every record in failed_jobs onto its original queue.
//   Monitor the queue immediately:
//     php artisan queue:work --queue=bookings --once
//   Watch the logs for new failures.

// STEP E — What queue:flush does and when it is appropriate
//   php artisan queue:flush  — permanently DELETES all records from failed_jobs.
//   Use this ONLY when:
//     - All failures are unrecoverable (e.g. test data, or the business
//       decision is to not process these bookings).
//     - You have already re-processed everything recoverable.
//   NEVER run queue:flush as a first response to a failure incident.
//   The failed_jobs table is your audit trail — it tells you what happened
//   and allows recovery. Flushing it destroys that history.
