SOLUTION 02 — Strategy Pattern: PricingContext, Runtime Selection, Open/Closed
===============================================================================
Focus: How the Strategy pattern works, runtime strategy selection, and testing

This file contains PHP comment skeletons only. No executable code.

──────────────────────────────────────────────
PART 1: Strategy pattern structure
──────────────────────────────────────────────

// The Strategy pattern defines a family of algorithms (pricing rules),
// encapsulates each one in a class, and makes them interchangeable
// The client (PricingContext) works with a strategy through its interface
// — it doesn't know or care which concrete strategy it holds

// The three roles:
// Strategy interface:   PricingStrategyInterface — defines calculate()
// Concrete strategies:  StandardPricing, GroupDiscountPricing, EarlyBirdPricing
// Context:              PricingContext — holds and delegates to a strategy

──────────────────────────────────────────────
PART 2: PricingContext — implementation approach
──────────────────────────────────────────────

// Private property: private PricingStrategyInterface $strategy

// setStrategy(PricingStrategyInterface $strategy): void
// Just assign: $this->strategy = $strategy
// No logic here — PricingContext is a "holder and delegator", not a decision-maker

// calculatePrice(Trip $trip, int $studentCount): float
// Delegate entirely: return $this->strategy->calculate($trip, $studentCount)
// Zero pricing logic in this method — just delegation

// Optional: constructor injection of a default strategy
// public function __construct(PricingStrategyInterface $strategy = null)
// If null, default to StandardPricing
// Avoids "strategy not set" errors when calculatePrice is called before setStrategy

// Why keep PricingContext logic-free?
// If PricingContext had conditionals, it would need to change every time
// a new strategy is added — violating Open/Closed Principle

──────────────────────────────────────────────
PART 3: StandardPricing — implementation approach
──────────────────────────────────────────────

// calculate(Trip $trip, int $studentCount): float
// return (float) ($trip->price_per_student * $studentCount)

// This is the baseline — no conditions, no special logic
// All other strategies compare against or fall back to this calculation

──────────────────────────────────────────────
PART 4: GroupDiscountPricing — implementation approach
──────────────────────────────────────────────

// calculate(Trip $trip, int $studentCount): float
// Condition: if $studentCount >= 20, apply 10% discount
// Calculation: $trip->price_per_student * $studentCount * 0.90
// Fallback: if $studentCount < 20, return standard price (no discount)
// return $trip->price_per_student * $studentCount  (same as StandardPricing)

// The fallback to standard price inside GroupDiscountPricing is correct
// because this class is responsible for the group discount rule including its threshold
// The caller selects this strategy — the strategy handles the "no discount" case internally

// Alternative: throw an exception or return null if threshold not met
// — NOT recommended: the caller shouldn't have to guard against null prices

PART 5: EarlyBirdPricing — implementation approach
──────────────────────────────────────────────

// calculate(Trip $trip, int $studentCount): float
// Need current date to compare against trip->departure_date
// $daysUntilDeparture = now()->diffInDays($trip->departure_date)
// Condition: if $daysUntilDeparture >= 60, apply 15% discount
// Calculation: $trip->price_per_student * $studentCount * 0.85
// Fallback: if < 60 days, return standard price

// Testing consideration:
// now() makes this hard to test — the calculation depends on the real current date
// Solution: inject a Carbon instance or a clock interface
// Or accept a $bookingDate parameter in calculate() and add it to the interface
// For this challenge, injecting $bookingDate into the constructor is the simplest fix:
// public function __construct(private \Carbon\Carbon $bookingDate) {}
// Then use $this->bookingDate->diffInDays($trip->departure_date) in calculate()
// Tests pass a fake date — no coupling to system time

PART 6: Strategy selection in BookingService
──────────────────────────────────────────────

// BookingService::create() needs to select the right strategy before calculating price
// The decision is based on two factors: studentCount and days until departure

// Decision logic (a private helper method: resolveStrategy(Trip $trip, int $count)):
// Check group discount first: if $count >= 20, use GroupDiscountPricing
// Check early bird second: if departure is 60+ days away, use EarlyBirdPricing
// Default: use StandardPricing

// Why group discount takes priority when both apply:
// — Business decision that must be documented explicitly
// — Group discount rewards the school for bringing more students (volume incentive)
// — Early bird is about timing; group size overrides it
// — Document this in a code comment so future developers don't inadvertently change it

──────────────────────────────────────────────
PART 8: Adding a new strategy — Open/Closed Principle in action
──────────────────────────────────────────────

// The team wants to add a LoyaltyPricing: schools with 10+ past bookings get 5% off

// What you need to create:
// — New class: LoyaltyPricing implements PricingStrategyInterface
// — calculate() method with the 5% discount logic
// — A way to check past booking count — inject BookingRepositoryInterface

// What you do NOT need to change:
// — PricingStrategyInterface (the interface is stable)
// — StandardPricing, GroupDiscountPricing, EarlyBirdPricing (not touched)
// — PricingContext (zero changes — it holds any strategy)
// — Only BookingService::resolveStrategy() gets one new condition

// This is Open/Closed: the system is open for extension (new class)
// but closed for modification (existing classes unchanged)

// Compare to the alternative (if/else chain in BookingService):
// adding LoyaltyPricing means editing the existing service method
// — risk of breaking existing pricing logic
// — the method grows without bound as more rules are added
// — each pricing rule is not independently testable
