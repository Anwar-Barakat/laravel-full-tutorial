SOLUTION 02 — Events vs Direct Calls, Testing in Isolation, Reference Generation
=================================================================================
Focus: Design decisions, testability rationale, and trade-offs

This file contains PHP comment skeletons only. No executable code.

──────────────────────────────────────────────
PART 1: Why fire an event instead of calling mail and notify directly?
──────────────────────────────────────────────

// Option A — Direct calls (what the god controller does)
// BookingService::create() calls Mail::to()->send() and $admin->notify() directly
// Works fine — but the service now "knows" about every downstream reaction

// Option B — Fire a BookingCreated event
// BookingService fires event(new BookingCreated($booking))
// BookingCreatedListener::handle() sends the email and notifications

// Why Option B is better for Tripz:
// — Open/Closed Principle: add a new reaction (e.g., webhook, SMS, analytics)
//   by adding a new Listener — no changes to BookingService
// — Loose coupling: the service does not know what happens after a booking is created
// — Queue-ability: listeners can be queued (implements ShouldQueue)
//   Heavy operations (email, SMS) move to background jobs automatically
// — Testability: in tests, fake the event dispatcher (Event::fake())
//   and assert events were dispatched without actually sending email

// When Option A is acceptable:
// — Small applications where the overhead of events is not justified
// — When you want to guarantee email is sent before returning the response
//   (queued listeners are async — the response may return before email is sent)
// — When the side effect is critical and must not fail silently

// For Tripz: use events + queued listeners for mail/notify, keep Option A for logging
// Log::info() is synchronous, cheap, and should not be deferred

PART 2: Testing CreateBookingAction in isolation
──────────────────────────────────────────────

// Because the Action has no HTTP dependency, you can call it directly in a test
// No need for $this->post('/api/bookings', [...]) HTTP round-trip

// Test setup:
// — Create a Trip with known max_capacity in the DB (or use a factory)
// — Create some existing Bookings to simulate partial capacity
// — Instantiate CreateBookingAction directly: new CreateBookingAction()
// — Call $action->execute($data)
// — Assert the returned Booking has the expected attributes

// Test: capacity exceeded
// — Set existing bookings so they fill the trip
// — Call execute() with student_count that would exceed max_capacity
// — Assert BookingCapacityException is thrown (use $this->expectException())

// Test: reference format
// — Call execute() with valid data
// — Assert $booking->reference matches the pattern /^TRP-[A-Z0-9]{8}$/

// Key point: none of these tests boot the HTTP kernel, hit routes, or need a token
// They are fast unit/integration tests that run independently of the web layer

PART 4: Booking reference generation — Action vs Model
──────────────────────────────────────────────

// Option A — Generate in CreateBookingAction
// The Action builds the full data array including the reference
// then calls Booking::create([...$data, 'reference' => $reference, ...])
// PRO: explicit, easy to test (just call execute() and check $booking->reference)
// PRO: the model stays passive — it doesn't generate its own data
// CON: if another code path creates a Booking directly (bypassing the Action),
//      the reference won't be set

// Option B — Generate in the Booking model via a boot() observer
// protected static function booted(): void
// static::creating(function (Booking $booking) {
//     $booking->reference = 'TRP-' . strtoupper(Str::random(8));
// })
// PRO: guaranteed on every creation regardless of code path
// PRO: model is self-contained — reference is always present
// CON: harder to test in isolation (requires actually triggering the model event)
// CON: the model now has a side-effectful boot hook — less transparent

// Recommendation for Tripz:
// Use Option B (model boot) as a safety net — ensures reference is always set
// The Action does NOT need to set it — the model handles it
// This way, if a factory or seeder creates Bookings directly, they still get a reference

PART 5: BookingCapacityException — design notes
──────────────────────────────────────────────

// Create app/Exceptions/BookingCapacityException.php
// Extend RuntimeException (or a custom DomainException base class)
// No HTTP knowledge — it is a pure domain exception

// The controller catches it and converts to an HTTP response:
// catch (BookingCapacityException $e) {
//     return response()->json(['error' => $e->getMessage()], 422)
// }

// Why not return a response directly from the Action?
// — Actions have no HTTP context — they should not produce HTTP responses
// — The same Action might be called from a CLI artisan command
//   where response()->json() makes no sense
// — Exceptions cross layer boundaries cleanly — each layer handles what it knows

// Alternative: use Laravel's Handler.php to map the exception globally
// Register a renderable for BookingCapacityException in Handler::register()
// Then the controller does not need the try/catch at all
// Useful if the exception can be thrown from multiple controllers

PART 7: The refactor in terms of line count
──────────────────────────────────────────────

// Before refactor: BookingController::store() — ~45 lines, 7 responsibilities
// After refactor:
// — CreateBookingRequest::rules()  — ~6 lines (validation only)
// — CreateBookingRequest::authorize() — 1 line
// — CreateBookingAction::execute() — ~15 lines (capacity + create)
// — BookingService::create() — ~12 lines (orchestration + side effects)
// — BookingController::store() — ~5 lines (delegate + respond)

// Total lines is slightly more — but each file has ONE reason to change
// A change to email logic touches only the Service (or the Listener)
// A change to validation rules touches only the FormRequest
// A change to the reference format touches only the Action (or Model boot)
