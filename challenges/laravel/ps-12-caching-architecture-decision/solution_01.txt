// SOLUTION 01 — Caching Architecture Decision: Driver Selection and TTL Strategy
// Topic: Designing a comprehensive caching architecture for Tripz at scale
// Approach: Redis choice reasoning, TTL decisions per endpoint, cache-aside pattern

// ============================================================
// QUESTION 1 — CACHE DRIVER SELECTION: WHY REDIS
// ============================================================

// At Tripz's scale (multiple web workers, 10k bookings/month), the file driver fails:
//
// File driver problems at scale:
//   - Each web worker has its own file system cache — no sharing between workers
//   - Worker A caches a trip list; Worker B doesn't see it — it re-queries the database
//   - Does not support cache tags (required for grouped invalidation)
//   - Reads/writes require file system I/O — slower than in-memory
//   - Cache files accumulate on disk and require manual cleanup
//
// Memcached problems:
//   - In-memory and fast (good), but lacks native support for cache tags in Laravel
//   - No persistence — restarts lose all cached data
//   - Limited data structure support (only strings)
//   - Less popular than Redis in the Laravel ecosystem
//
// Redis advantages for Tripz:
//   - Shared across all web workers and server instances — one cache store for all
//   - Supports cache tags natively through set operations
//   - Supports Cache::lock() for stampede prevention (atomic SETNX)
//   - Supports Cache::flexible() (stale-while-revalidate) via atomic operations
//   - Key expiration is handled at the data structure level (no garbage collection needed)
//   - Pub/sub support for future real-time features
//   - Laravel Horizon (for queues) also uses Redis — one service for both
//
// Conclusion: Use Redis in staging and production
//   CACHE_DRIVER=redis in .env
//   Use CACHE_DRIVER=array in tests (in-memory, supports tags, fast, no infrastructure needed)

// ============================================================
// QUESTION 2 — TTL STRATEGY PER ENDPOINT
// ============================================================

// GET /api/trips — public list
//   TTL: 1 hour (3600 seconds)
//   Reasoning: Trip listings change rarely (new trips are added occasionally, not continuously)
//   At 1,000 requests/day, caching for 1 hour reduces 1,000 queries to ~24 queries/day
//   Cache key: tripz:trips:list:{page}:{filters_hash}
//   Note: include pagination and filter params in the key to avoid serving wrong pages
//
// GET /api/trips/{id} — public trip detail
//   TTL: 1 hour (3600 seconds)
//   Reasoning: Individual trip data (name, price, capacity, dates) rarely changes
//   At 3,000 requests/day per trip * 500 trips, caching is critical
//   Cache key: tripz:trips:{id}
//   Invalidate on: Trip saved/deleted event
//
// GET /api/bookings — private per-user
//   TTL: NO CACHING recommended
//   Reasoning: Booking lists are user/role-specific and change frequently
//   Caching would require per-user cache keys (tripz:bookings:user:{userId})
//   The data changes every time a booking is created/updated/deleted
//   With 500 requests/day, the query cost is acceptable without caching
//   Caching private, sensitive financial data in Redis requires careful key scoping
//   A misconfigured key could serve user A's bookings to user B
//   Risk-to-reward ratio does not justify caching at this scale
//
// GET /api/dashboard/stats — admin only
//   TTL: 10 minutes (600 seconds)
//   Reasoning: 30-second queries must be cached to be usable
//   Admin-only endpoint — same cached value is valid for all admins (no per-user data)
//   10 minutes balances freshness with performance
//   Cache key: tripz:dashboard:stats (shared across all admin users)
//   Invalidate on: Booking created/updated/deleted events

// ============================================================
// THE CACHE-ASIDE PATTERN (READ STRATEGY)
// ============================================================

// This is the pattern used for all application-level caching in this solution
//
// Read flow:
//   1. Request arrives for GET /api/trips
//   2. Check cache: Cache::get('tripz:trips:list:page1')
//   3. Cache HIT → return cached data immediately (0 database queries)
//   4. Cache MISS → query database: Trip::with('school')->paginate(20)
//   5. Store result in cache: Cache::put('tripz:trips:list:page1', $trips, 3600)
//   6. Return the freshly-queried data
//
// Write flow (cache invalidation):
//   1. Admin updates a trip
//   2. Trip observer fires: TripObserver::updated($trip)
//   3. Observer clears: Cache::tags(['trips'])->flush()
//   4. Next request for any trip endpoint hits a cache miss and re-warms the cache
//
// This pattern is called "lazy loading" — cache is populated on demand, not upfront
// The first request after a cache miss is slow; subsequent requests are fast
// Pre-warming (a scheduled job that runs queries proactively) eliminates cold-start delay

// ============================================================
// CACHE KEY DESIGN — HANDLING PAGINATION AND FILTERS
// ============================================================

// A trip list with 500 trips needs pagination
// GET /api/trips?page=2&destination=london produces different data than GET /api/trips?page=1
// The cache key must capture all query-affecting parameters
//
// Approach: hash the query parameters and include in the key
//   $filterHash = md5(json_encode($request->query())); // hash of all query params
//   $key = "tripz:trips:list:{$filterHash}";
//
// Or build the key explicitly:
//   $key = "tripz:trips:list:page:{$page}:dest:{$destination}:sort:{$sort}";
//
// The hash approach handles any combination of filters without managing each param
// The explicit approach is more readable but requires updating if new filters are added
//
// Note: if filters include user IDs or session tokens, NEVER include those in a shared cache key
